Chrome's HTML-in-Canvas API
is here — and it's wild
Google's experimental layoutsubtree attribute lets browsers render live DOM elements directly into a WebGL texture — enabling a new class of spatial UIs.
Read More
Deep Dive
What does it mean for the web?
For years, developers have hacked around the browser's compositing model to get HTML content into WebGL. Libraries like html2canvas used screenshot tricks, while CSS3DRenderer faked depth with matrix transforms. Neither approach gave you true WebGL integration.
The new layoutsubtree canvas attribute changes that. Child DOM nodes are rasterized by the browser's own layout engine and exposed to the canvas paint event — meaning hover states, CSS animations, and interactive elements all work inside the texture.

Under The Hood
How HTMLTexture actually works
At its core, THREE.HTMLTexture wraps a canvas element and hooks into its onpaint callback — a single-slot property that fires whenever the browser's layout engine decides the subtree needs re-rasterizing. When the callback fires, Three.js marks the texture as dirty and uploads the new bitmap to the GPU on the next render frame.
One critical constraint: onpaint is a single-slot property, meaning only one listener can be registered per canvas. This is why every live window requires its own source canvas — sharing one canvas across multiple textures means only the last registered listener fires.
The multi-canvas architecture
A production implementation would likely maintain a pool of off-screen source canvases, each rasterizing one window's DOM subtree. The visible WebGL canvas composites all of them into the final scene. Because each source canvas is independent, windows can animate, update content, and scroll without affecting one another.
Depth ordering is handled entirely by mesh position.z values — no CSS z-index, no DOM reordering. Clicking a window moves it to the top of the stack by assigning it the highest z-slot; all other windows shift down by one slot to keep the range bounded.
Performance
What's fast, what's not
Static windows are essentially free — the rasterized bitmap sits on the GPU and the render loop just draws it every frame with no CPU cost. Dynamic content (CSS transitions, animated text, live data) triggers repaints proportional to the canvas size. A 3440×1440 desktop chrome canvas repainting at 60fps is expensive; a 3440×39 menubar canvas is trivial.
The practical rule: keep frequently-updating elements in small source canvases. Static chrome — wallpaper, dock, trash — can share a single large canvas that only repaints when explicitly requested (zone drag previews, for instance).

What's Next
The road to full interactivity
The remaining challenge is routing input events back into the source canvas DOM. Because all pointer events land on the WebGL canvas, a raycaster must map screen coordinates to UV coordinates, then convert UV to window-local pixels, and finally dispatch synthetic events to the element under the cursor via elementFromPoint.
Scrolling is already achievable by caching scrollable element references at init time and applying scrollTop deltas on wheel events. Hover states and clicks require the full synthetic event pipeline — expected to land in a future Chrome Canary build with improved layoutsubtree hit-testing support.
